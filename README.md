# eBPF in Go: Observability for AI-Generated Services

A hands-on workshop companion repository for the blog post:

👉 **[eBPF in Go: Observability for AI-Generated Services](https://medium.com/p/9aae7573b823?postPublishedType=initial)**

> *From blind debugging to kernel-level visibility in 10 minutes*

This project demonstrates how to use **eBPF** (extended Berkeley Packet Filter) from **Go** to add kernel-level observability to services — including those generated by AI — where traditional application-level instrumentation is missing or unreliable.

## 🎯 What This Project Does

This program attaches a small eBPF program to the `sys_enter_openat` tracepoint in the Linux kernel. Whenever any process on the system calls `openat()` (i.e. opens a file), the kernel:

1. Captures the **PID**, **timestamp**, and **process name (`comm`)**.
2. Sends the event to a **perf event map** in kernel space.
3. The Go userspace program reads from that map via a **perf reader**.
4. Prints the event to stdout in real time.

In the context of the blog post, this technique is applied to AI-generated services where you don't fully trust or understand the code path — eBPF gives you visibility at the syscall layer, independent of what the application is doing (or claiming to do).

## 🚀 Getting Started

### Prerequisites

- **Linux kernel** ≥ 4.17 (with `CONFIG_BPF=y`, `CONFIG_BPF_SYSCALL=y`, `CONFIG_BPF_JIT=y`)
- **Go** ≥ 1.21 (tested with 1.26.4)
- **clang** and **LLVM** toolchain (for regenerating the eBPF object)
- **Root privileges** (or appropriate `CAP_BPF` / `CAP_PERFMON` capabilities) to load eBPF programs

### Build & Run

```bash
# Clone the repository
git clone https://github.com/cheikh2shift/ebpf-workshop.git
cd ebpf-workshop

# (Optional) Regenerate the BPF object if you modify bpf/trace.c
go generate ./...

# Run with root privileges
sudo go run main.go
```

You should see output like:

```
Observing file opens... Press Ctrl+C to exit.
PID 1234 (systemd) opened a file
PID 5678 (my-ai-service) opened a file
...
```

Open another terminal and run any command (`ls`, `cat`, etc.) — you'll see the corresponding `openat` events appear in real time.

## 🔬 How It Works

1. **`go:generate` directive** in `main.go` invokes [`bpf2go`](https://github.com/cilium/ebpf/tree/main/cmd/bpf2go), which compiles `bpf/trace.c` into a BPF ELF object and generates Go bindings.
2. **`main.go`** uses [`github.com/cilium/ebpf`](https://github.com/cilium/ebpf) to:
   - Remove the `RLIMIT_MEMLOCK` so the process can pin BPF maps.
   - Load the embedded BPF object.
   - Attach it to the `syscalls/sys_enter_openat` tracepoint.
   - Open a perf reader on the `events` map and print every event.
3. **`bpf/trace.c`** defines:
   - `trace_openat()` — the kernel-side handler invoked on every `openat` syscall.
   - `events` — a perf event array map used to push data to userspace.

For a deeper explanation of *why* and *when* to use this pattern, see the [blog post](https://medium.com/p/9aae7573b823?postPublishedType=initial).

## 🧠 Why eBPF for AI-Generated Services?

AI-generated code can be opaque, over-engineered, or quietly call into libraries you didn't intend. Traditional observability (logs, metrics, APM) only tells you what the code *chose* to report. **eBPF observes the kernel directly**, so you get an honest view of syscalls, network calls, file I/O — independent of the application's instrumentation. This repo is a minimal but powerful example of that principle.
